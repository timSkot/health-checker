# Health Checker

Консольная утилита на Go для параллельной проверки доступности списка сайтов — построена на worker pool с ограниченным числом горутин, таймаутами через `context` и потокобезопасным сбором статистики.

## Что делает

Читает список URL из файла, проверяет каждый через HTTP GET, и для каждого выводит статус-код, время отклика и ошибку (если сайт недоступен или не ответил вовремя). Проверки выполняются параллельно, ограниченным числом воркеров, а не последовательно и не безлимитным числом горутин.

## Стек

- Go 1.22+, только стандартная библиотека (`net/http`, `context`, `sync`, `bufio`)

## Архитектура конкурентности

- **Worker pool**: фиксированное число горутин (`numWorkers`) разбирают задачи из общего канала `jobs`, вместо горутины на каждый URL — контролируемая нагрузка вместо неограниченного параллелизма.
- **Каналы** `jobs` и `results` — передача задач воркерам и сбор результатов обратно, без разделяемой памяти.
- **`context.WithTimeout`** — каждый HTTP-запрос ограничен по времени (3 секунды по умолчанию); недоступные или медленные сайты не блокируют всю проверку.
- **`sync.WaitGroup`** — ожидание завершения всех воркеров перед закрытием канала результатов.
- **`sync.Mutex`** — защита общего счётчика успешных проверок от гонки данных при одновременной записи из нескольких воркеров.

## Запуск

Список URL — по одному на строку в `urls.txt`:

```
https://google.com
https://github.com
https://golang.org
```

Запуск:

```bash
go run main.go
```

Проверка на гонки данных (полезно при внесении изменений в конкурентную часть):

```bash
go run -race main.go
```

## Пример вывода

```
https://github.com - status: 200, duration: 326.352542ms, err: <nil>
https://golang.org - status: 200, duration: 913.043125ms, err: <nil>
https://google.com - status: 200, duration: 1.021484792s, err: <nil>
Total time: 1.021672125s
Successful checks: 3
```

Обратите внимание — порядок вывода соответствует порядку **завершения** запросов, а не порядку в файле `urls.txt`: результаты приходят по мере готовности от параллельно работающих воркеров.

## Почему worker pool, а не горутина на каждый URL

При большом списке адресов (сотни/тысячи) запуск горутины на каждый URL создал бы такое же количество одновременных исходящих соединений — риск исчерпать сетевые ресурсы или получить блокировку от целевых серверов за агрессивный трафик. Worker pool явно ограничивает, сколько проверок выполняется параллельно в любой момент времени, независимо от размера списка.
